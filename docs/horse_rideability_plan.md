# Rideable Horse Conversion: Oblivion CREA → Skyrim Mountable Actor

Feasibility + implementation plan for making converted Oblivion horses behave like
vanilla Skyrim horses (player can approach and mount them). This sits ON TOP of the
existing creature pipeline (see [docs/creature_conversion.md](creature_conversion.md)) —
horses already convert through that path as ordinary quadrupeds (locomotion, combat,
death). This doc covers only the DELTA needed for the ridden state.

**Verified against**: `references/Skyrim.esm/RACE.txt` (HorseRace 000131FD),
`references/Skyrim.esm/NPC_.txt` (WhiterunPlayerHorse + 4 other city horses),
`references/Skyrim.esm/PACK.txt` (PlayerHorseWaitInPaddock 00109AB2),
`references/xEdit/Core/wbDefinitionsTES5.pas` (RACE DATA struct), `references/Skyrim
Animations/meshes/actors/horse/` + `.../actors/character/behaviors/horsebehavior.hkx`
(extracted LE animation archive), `references/Skyrim Meshes/meshes/actors/horse/character
assets/skeleton.nif`. Oblivion side: `export/Oblivion.esm/CREA.txt` (54 horse CREA
records, e.g. `HorseBayClaudeMaric`, `MQHorseJauffre`, `HorseBlackCourier1-4`).

---

## 1. There is no "IsMount" flag

Checked every RACE flag bit (`wbRACE_DATAFlags01`, `Flags 2`) and the horse's KWDA
(`00013798`=animal, `00013795`=creature, plus two unrelated keywords
`MagicNoReanimate`/`ImmuneParalysis` — NOT mount-specific). **Mountability is not a
single switch.** It's assembled from four independent pieces, verified below.

---

## 2. What actually makes a Skyrim actor mountable

### 2.1 RACE "Mount Data" struct (easy — data only)

`wbDefinitionsTES5.pas:9430`, gated behind form-version ≥29 (our target FormVersion=44
already qualifies):

```
Mount Offset X/Y/Z        (vanilla horse default ~ -63.48, 0, 0)
Dismount Offset X/Y/Z     (vanilla default ~ -50, 0, 65)
Mount Camera Offset X/Y/Z (vanilla default ~ 0, -300, 0)
```

These are relative to the horse's own root — not per-mesh, not derived from geometry.
Also `Flags 2` bit `0x10` = "Allow Mounted Combat" (optional, vanilla horse leaves it
unset — the player can't fight from horseback in vanilla Skyrim either, so we should
match that rather than trying to add it).

**Action**: `tes5_import/creature_races.py` RACE builder gets a `mount_data` field,
populated with the vanilla horse constants above for any creature folder flagged as a
mount (see §3.1 — driven by a manifest flag, not creature-name string matching).

### 2.2 A dedicated behavior graph pair (the real work)

Confirmed via the extracted LE animation archive — Bethesda ships **two** separate
graphs, not a horse variant of `quadrupedbehavior.hkx`:

| File | Role | Size (LE) |
|---|---|---|
| `meshes/actors/horse/behaviors/horsebehavior.hkx` | Horse-side: locomotion + mount/dismount poses + paired-animation sync + jump states | 83,408 B |
| `meshes/actors/character/behaviors/horsebehavior.hkx` | **Rider-side**: runs on the PLAYER's own skeleton while mounted, drives `animations/horse_rider/*` in sync with the horse | 76,416 B |

For scale: this is the same order of magnitude as the quadruped graphs already
reverse-engineered and generated by `asset_convert/hkx_behavior.py` (bear 94,608 B,
canine/wolf 133,344 B, deer 81,728 B) — a new template of comparable complexity, not a
new category of problem.

The horse-side graph is a superset of the quadruped template: everything
`hkx_behavior.py` already generates (locomotion blend, turn states, attack/recoil/
stagger/death, `iState_*`/MOVT contract, `BSSpeedSamplerModifier`) PLUS:
- `mountStart`/`mountStop` (or vanilla's actual event names — dump via `hkxcmd
  convert -v:XML` before implementing) entering/exiting a "Ridden" branch that
  suppresses independent AI locomotion in favor of player-driven input.
- Paired mount/dismount single-play states keyed to the rider's matching states
  (paired-animation contract: both graphs must reach their synced pose at the same
  wall-clock time — check `horse_paired_mount.hkx`/`horse_paired_dismount.hkx` timing
  data in `animationdata/boundanims/anims_horseproject.txt`).
- Jump states (`horse_jumpstart/jumpfall/jumpland[_sprint]`) — Oblivion horses have no
  jump animations or jump mechanic at all; this can be scoped out of v1 (horse simply
  can't jump gates/fences, matching Oblivion's own riding behavior).

The rider-side graph is new to this project in kind (nothing else generates a graph
that runs on the PLAYER character), but structurally it's a strict subset: no combat,
no death, no locomotion — just mount/dismount poses + a ride-loop blend synced to the
horse's `Speed` variable.

### 2.3 Rider animations (no Oblivion source — must be sourced from vanilla)

Oblivion's mounted riding is a hardcoded engine hack: the player actor is disabled and
its position is written directly (`SetPos`-equivalent) every frame relative to the
horse bone, with the player camera simply following — **there is no player-mounted
animation of any kind in TES4**, and Oblivion horses' own KF corpus has no
paired-mount/dismount/ride-loop clips either (Oblivion horse KFs are ordinary
forward/idle/turn walk cycles, structurally identical to every other quadruped —
confirmed against the same `export/Oblivion.esm/meshes/creatures/horse/` inventory the
existing creature pipeline already reads).

This means, unlike every other piece of the creature pipeline (which is a *faithful
port* of Oblivion source data), **the mount/dismount/ride-loop animations have no TES4
equivalent to convert.** Two options:
- **(A) Reuse vanilla Skyrim's own horse + horse_rider clips verbatim** (already
  Havok-format, zero KF-decode/B-spline work, zero animation-conversion risk) — the
  pragmatic v1 path, matching how NIF/texture assets are already sourced from vanilla
  SSE BSAs per `asset_convert/skyrim_assets.py` (never from `references/`, always the
  auto-extraction path).
- (B) Author new poses/blend the Oblivion horse's existing walk/idle cycle into a
  seated approximation — much higher effort, cosmetic gain only, not worth it for v1.

**Recommendation: (A).** The horse mesh/skeleton is still the CONVERTED Oblivion horse
(faithful port, per project philosophy); only the mount-state ANIMATION DATA is vanilla
Skyrim's, because Oblivion supplies none. This is a narrower exception than reusing a
donor behavior graph wholesale (§"Why NOT donor behavior graphs" in
creature_conversion.md doesn't apply here — that section is about locomotion/combat,
which we DO generate faithfully from Oblivion clips; riding is the one sub-system with
no source data at all).

### 2.4 Script/record plumbing (small)

- **NPC_**: `RNAM` → the horse's generated RACE (already how `convert_CREA` works);
  no special ACBS flags found on vanilla `WhiterunPlayerHorse` beyond ordinary NPC
  flags (`0x00040018`) — nothing mount-specific here either.
- **AI package**: vanilla `PlayerHorseWaitInPaddock` (00109AB2) is an ordinary
  wander/wait-in-location package — dumped and confirmed there is NO special
  "mountable" package type. Mounting is purely a runtime `Actor.Mount()`/engine action,
  not AI-package-gated. Existing `tes5_import/packages.py` creature-package
  substitution (DefaultMasterPackageCreature) is sufficient; no new package logic
  needed.
- **Papyrus/engine hookup**: player interacts via the standard "Mount" activate prompt,
  which the engine derives from the RACE having valid Mount Data + the actor not being
  hostile/dead — no script authoring needed on our side for the base case.

## 3. Implementation plan

### Step 1 — Manifest: identify mount-capable creature folders
Extend `tools/creature_inventory.py` (or the `creature_projects.json` contract) with a
`is_mount: bool` flag, set for the `horse` folder (matched by clip inventory shape —
check for paired/rider-adjacent clip absence rather than hardcoding the string "horse",
since Nehrim/other plugins may ship differently-named horse-equivalents or additional
mountable creatures — e.g. detect via a config allowlist keyed on skeleton bone census
matching the vanilla horse, falling back to folder-name heuristic like the existing
`_FOLDER_KEYWORDS` animal-tagging in `creature_races.py:107`).

### Step 2 — RACE Mount Data (creature_races.py)
Add `mount_data` struct emission gated on `is_mount`, using the vanilla HorseRace
constants (§2.1) as the only values we ship — Oblivion has no equivalent field to pull
per-plugin values from, so this is intentionally a constant, not derived data (same
category of exception as horse animations, §2.3).

### Step 3 — Rider-side vanilla asset passthrough
Add a `mount_assets` stage to `asset_convert/creature_pipeline.py` (horse-only): pull
vanilla `actors/character/behaviors/horsebehavior.hkx` +
`animations/horse_rider/*.hkx` + `actors/horse/behaviors/horsebehavior.hkx` mount/
dismount/ride-loop clips via `asset_convert/skyrim_assets.py` (cache in
`export/skyrim_assets/`, same contract as every other vanilla-asset pull in this
project — never touch `references/` at runtime). No conversion needed — these are
already SSE-loadable Havok/NIF.

### Step 4 — Horse behavior graph: extend the quadruped template
In `asset_convert/hkx_behavior.py`, add a `mount` branch to `build_behavior_xml` (only
emitted when `is_mount`): a `Ridden` state entered/exited by whatever event names
vanilla actually uses (dump `horsebehavior.hkx` via `hkxcmd convert -v:XML` FIRST —
don't guess event names, same rule as every other behavior-graph fact in this project)
wrapping the horse's own locomotion states but driven by the rider's input variables
instead of AI `Speed`/`Direction`. Reuse every existing mechanism
(`BSSpeedSamplerModifier`, `iState_*`/MOVT contract, nested SM topology rules from the
2026-07-16 head-whipping fix) — the Ridden state is additive, not a replacement
graph.

### Step 5 — Verification
- `tools/creature_vanilla_ab.py` A/B: our converted horse race/records + vanilla horse
  behavior/animation assets, to isolate record-side correctness from graph correctness
  (same bisection method used throughout creature_conversion.md).
- In-game: spawn a converted horse, `player.placeatme`, attempt mount via activate
  prompt; verify camera offset, dismount, and that ordinary (non-ridden) locomotion/
  combat/death — already working via the base creature pipeline — is unaffected.
- Explicitly OUT of scope for v1 (document as known limitation, matching Oblivion's own
  behavior): jump-over-obstacles, mounted combat, mounted fast-travel prompts tied to
  Skyrim's stable/city-horse ownership system (Oblivion has no per-city owned-horse
  concept — every converted horse is a plain CREA, so "buy a horse at the stable"
  UX would need its own separate design pass, not covered here).

## 4. Open questions (resolve during implementation)

1. Exact vanilla event names for mount/dismount/ridden-state transitions — must dump
   `horsebehavior.hkx` XML rather than assume names by analogy to quadruped events.
2. Whether the rider-side graph can be grafted onto the player's EXISTING behavior
   graph as a sub-state (likely — vanilla does exactly this, since the player has only
   one graph total) or needs separate handling in whatever governs the player actor's
   graph in this pipeline (the project currently has no "player behavior" conversion
   surface at all — this may be the first).
3. Whether non-horse Oblivion creatures should ever be mount-flagged (e.g. Nehrim or
   other plugins may add rideable variants) — the manifest-driven `is_mount` flag in
   Step 1 should stay data-driven rather than hardcoded to unblock this later without
   code changes.
